# Local HTTPS and Low-Cost Demo Deployment

This repository keeps the backend and frontend on HTTP inside Docker. HTTPS is terminated at Nginx, which exposes:

- `https://localhost/` or `https://your-domain.com/` for the frontend.
- `https://localhost/api/*` or `https://your-domain.com/api/*` for Spring Boot.

This makes the browser use a single origin and avoids CORS issues. Login remains `POST /api/auth/login`.

## Environment Variable Matrix by Mode

Use different values for `FRONTEND_API_URL` and `FRONTEND_CONFIRMATION_URL` depending on the public entry point the browser will use.

| Mode | Compose | Entry URL | `FRONTEND_API_URL` | `FRONTEND_CONFIRMATION_URL` |
|---|---|---|---|---|
| Local HTTP | `compose.yaml` | `http://localhost:4200` | `http://localhost:8080/api` | `http://localhost:4200/confirmar-email` |
| Local HTTPS with Nginx | `compose.yaml` + `compose.https-local.yaml` | `https://localhost` | `https://localhost/api` | `https://localhost/confirmar-email` |
| Public demo with Nginx | `compose.yaml` + `compose.demo.yaml` | `https://demo.example.com` | `https://demo.example.com/api` | `https://demo.example.com/confirmar-email` |

To avoid editing the same `.env` file every time, you can keep separate files and pass them with `--env-file`:

```bash
docker compose --env-file .env.local-http up -d

docker compose --env-file .env.local-https \
  -f compose.yaml \
  -f compose.https-local.yaml \
  up -d

docker compose --env-file .env.demo \
  -f compose.yaml \
  -f compose.demo.yaml \
  up -d
```

## Local Development with a Self-Signed Certificate

1. Generate the local certificate:

   ```bash
   sh infra/scripts/generate-local-certs.sh
   ```

2. Start the application with Nginx:

   ```bash
   FRONTEND_API_URL=https://localhost/api \
   FRONTEND_CONFIRMATION_URL=https://localhost/confirmar-email \
   docker compose -f compose.yaml -f compose.https-local.yaml up -d
   ```

3. Open:

   ```text
   https://localhost/
   ```

The browser will warn that the certificate is self-signed. To avoid the warning, replace the files in `infra/nginx/certs/local/` with certificates generated by `mkcert`.

## Public Demo with Let's Encrypt

Use `infra/scripts/init-letsencrypt.sh` only on a public VM or EC2 instance where your DNS name already points to the server and ports `80` and `443` are open.

The script:

- Creates a temporary certificate so Nginx can start.
- Starts Nginx with `compose.demo.yaml`.
- Requests a real Let's Encrypt certificate through Certbot using the HTTP challenge.
- Stores the certificate material under `infra/nginx/certbot/`.
- Reloads Nginx after the real certificate is available.

Test first against Let's Encrypt staging to avoid consuming production rate limits:

```bash
sh infra/scripts/init-letsencrypt.sh demo.example.com admin@example.com --staging
```

If staging works, request the real certificate:

```bash
sh infra/scripts/init-letsencrypt.sh demo.example.com admin@example.com
```

Use your real domain and email. `demo.example.com` and `admin@example.com` are documentation placeholders and will not work for a real certificate.

## Certificate Renewal

Run periodically on the VM:

```bash
docker compose -f compose.yaml -f compose.demo.yaml run --rm certbot renew
docker compose -f compose.yaml -f compose.demo.yaml exec nginx nginx -s reload
```

For a simple demo, you can run it from `cron` once per day.

## Ports and Exposure

Nginx exposes `80` and `443`. The base `compose.yaml` also exposes the frontend and backend to make development easier. For a more locked-down public demo, remove or restrict the direct `frontend`, `backend`, and `postgres` ports on the server.
